

# **TAVR-Analytics Workflow 插件：模块一详细设计文档**

## **一、引言与架构背景**

### **1.1. 文档目的**

本文档为“TAVR-Analytics Workflow”3D Slicer插件的**模块一：数据导入与场景准备**提供完整、详尽的技术设计规范。其旨在作为软件开发团队实施该模块功能的主要技术蓝图与实现指南，确保开发过程的准确性、一致性与高效性。本文档将明确定义模块的功能范围、数据结构、用户界面（UI）、工作流程逻辑、错误处理机制以及与其他模块的应用程序接口（API），从而为构建一个稳定、可靠且符合临床需求的软件奠定坚实基础。

### **1.2. 模块范围与职责**

模块一是整个TAVR术后分析工作流的 foundational entry point（基础入口点）。其功能边界被严格界定，核心职责包括：

* **数据加载与验证**：提供一个标准化的用户界面，用于引导用户选择并加载经导管主动脉瓣置换术（TAVR）术后的四维（4D）心脏CT DICOM数据序列。  
* **数据格式解析**：利用3D Slicer的底层架构，确保将加载的时间序列数据正确地解析并实例化为场景中的vtkMRMLSequenceNode对象，这是进行任何动态分析的前提 1。  
* **元数据自动填充**：在数据加载后，自动解析DICOM元数据（metadata），提取患者基本信息与检查信息，并用以填充内部数据模型，实现与“杭州方案术后CT核心实验室评估表”的高度对应 1。  
* **关键信息配置**：强制要求用户输入无法从DICOM中直接获取、但对后续分析至关重要的配置信息，特别是植入的TAVR**瓣膜品牌**与**瓣膜型号**。  
* **心动周期管理**：提供一个交互式的可视化工具，允许用户在整个心动周期内进行流畅的导航与审阅。  
* **关键时相标记与持久化**：使用户能够精确识别、标记并持久化存储对于临床评估至关重要的心脏周期时相（例如，舒张末期、收缩末期），以供后续的分割、测量及分析模块调用 1。

### **1.3. 架构定位**

本模块将作为3D Slicer框架下的一个脚本化Python模块（Scripted Python Module）进行开发。其技术栈将深度依赖于3D Slicer的核心组件：利用Slicer Python API进行场景管理（MRML树）、节点操作和逻辑控制；利用Qt框架（通过Slicer封装的qt或PySide库）构建用户界面；并利用VTK（Visualization Toolkit）库处理底层的图像数据表示与几何计算。

在整个插件的宏观架构中，模块一扮演着“数据提供者”和“分析上下文设定者”的关键角色。它负责初始化分析会话，并将所有加载的数据、用户输入的配置以及标记的关键时相信息，通过一个中央化的会话管理对象（Session Management Object）暴露给插件中的其他模块（如模块二：分割与标志点定义，模块三：自动化测量等），从而形成一个数据驱动的、逻辑连贯的集成化分析环境。

## **二、数据模型与状态管理**

### **2.1. 核心数据结构**

为了确保整个插件会话期间数据的一致性、完整性和可访问性，模块一将设计并实例化一个中央化的状态管理对象。该对象在插件启动时创建，并以单例（Singleton）模式存在，确保所有模块访问的是同一份数据状态。

#### **TAVRStudySession 类**

这是插件的主逻辑控制器和核心状态容器。它将负责管理整个分析会y话的生命周期，并持有对关键数据和节点的引用。其核心职责包括：

* 持有对加载到Slicer场景中的关键MRML节点的引用，如vtkMRMLSequenceNode和vtkMRMLSequenceBrowserNode。  
* 持有一个PatientData对象的实例，用于封装所有患者和研究相关信息。  
* 维护一个数据结构，用于存储用户标记的关键心动周期时相信息。

#### **PatientData 类**

这是一个专用的数据类（Data Class），其设计目标是精确地封装“杭州方案术后CT核心实验室评估表”1 中“一、基本情况”部分所需的所有信息。这种一对一的映射关系确保了从数据采集到最终报告生成的无缝对接。

#### **瓣膜特异性逻辑的架构设计**

在设计数据模型时，必须前瞻性地考虑一项对整个插件功能至关重要的技术要求。临床评估表单 1 的脚注中明确指出：“各瓣膜测量对照点”，这意味着不同品牌和型号的瓣膜（如Medtronic Evolut, Edwards SAPIEN3, VenusA等）其关键解剖平面的定义方式完全不同。例如，Evolut瓣膜的

nadir level（瓣叶窦底平面）定义在“1.5个菱形格”处，而SAPIEN3瓣膜则定义在“底部往上0.5个菱形格”处。

这一临床规范直接决定了插件后续模块（尤其是模块二和模块三）的行为。因此，用户在模块一中选择的“瓣膜品牌”和“瓣膜型号”绝非简单的描述性信息，而是一个关键的**配置参数**，它将动态地决定后续应采用何种测量算法和几何参考框架。

为了应对这一挑战，插件的架构必须采纳一种灵活的设计模式，如策略模式（Strategy Pattern）。TAVRStudySession对象必须将用户选择的瓣膜型号作为一个关键属性进行存储（例如，一个字典：{'brand': 'Medtronic', 'model': 'Evolut R/PRO'}）。模块一的API必须提供一个公共的获取方法（getter function），如get\_selected\_valve()，以便其他模块能够查询此配置。下游模块在执行测量前，会先调用此方法，然后根据返回的瓣膜型号，动态加载或选择正确的测量逻辑。这一设计确保了插件的临床准确性和未来的可扩展性，能够轻松地支持新的瓣膜型号，而无需重构核心代码。同时，UI设计必须强制用户完成此项选择，否则无法进入后续分析步骤，从而从根源上避免了因配置错误导致的测量谬误。

### **2.2. 场景与节点管理**

模块一在成功加载数据后，将在3D Slicer的MRML（Medical Reality Modeling Language）场景中创建并管理两个核心节点：

* **vtkMRMLSequenceNode**: 此节点用于存储完整的4D心脏CT图像数据。它将序列中的每一帧（一个3D容积）与一个索引值和时间戳关联起来，是所有时间相关分析的基础。  
* **vtkMRMLSequenceBrowserNode**: 此节点作为vtkMRMLSequenceNode的控制器。它负责管理序列的播放、同步和当前帧的选择。模块一的用户界面（如心动周期滑块）将直接与此节点交互，以控制在Slicer视图中显示哪一帧图像。

TAVRStudySession对象将持久化存储这两个节点的唯一ID（NodeID），确保插件的任何部分在需要时都能通过Slicer的场景单例（slicer.mrmlScene）可靠地获取到这些核心节点，从而实现模块间的解耦和稳健通信。

## **三、组件级设计：数据加载与验证**

### **3.1. 用户界面规格**

插件启动后，将呈现一个专用的数据加载与配置对话框。此对话框是用户与插件交互的第一个触点，其设计必须清晰、直观且功能完备。

* **UI布局**: 对话框将采用垂直布局，自上而下引导用户完成操作。  
* **核心组件**:  
  1. **数据加载按钮**: 一个标签为“加载4D DICOM序列”的QPushButton。点击后将调用Slicer原生的DICOMBrowser，允许用户选择一个DICOM序列。  
  2. **序列信息显示**: 一个只读的QLineEdit或QLabel，用于显示当前已成功加载的序列名称或ID，为用户提供明确的反馈。  
  3. **患者信息表**: 一个QTableView或一组QFormLayout布局的控件，用于展示和编辑PatientData对象中的信息。所有从DICOM自动填充的字段均可编辑，以处理元数据不完整或错误的情况。  
  4. **瓣膜选择器**:  
     * 一个标签为“瓣膜品牌”的QComboBox（下拉菜单），选项从外部配置文件动态加载。此项为必选项。  
     * 一个标签为“瓣膜型号”的QComboBox，其内容会根据“瓣膜品牌”的选择而动态更新。此项同样为必选项。  
     * 这两个控件将通过视觉提示（如星号\*或高亮边框）明确其强制性。  
  5. **操作按钮**:  
     * 一个“确认并继续”的QPushButton。此按钮在初始状态下为禁用（disabled）。只有当一个有效的4D序列被加载，并且用户已选择瓣膜品牌和型号后，此按钮才会被激活。  
     * 一个“取消”的QPushButton，用于关闭对话框并中止工作流。

### **3.2. DICOM解析与元数据映射逻辑**

数据加载的核心逻辑封装在“加载4D DICOM序列”按钮的点击事件处理器中。

1. **调用Slicer DICOM模块**: 该处理器将调用Slicer的DICOMUtils.loadDICOMBySeries函数，并将examine=True以允许用户在加载前预览和选择序列。  
2. **数据类型验证**: 加载完成后，脚本将立即检查返回的节点列表。它必须验证列表中包含一个且仅一个slicer.vtkMRMLSequenceNode的实例。如果不是，则认为加载失败，并触发相应的错误处理（见第五部分）。  
3. **元数据提取与填充**: 验证成功后，脚本将访问序列中第一帧图像的DICOM头文件。通过slicer.dicomDatabase或直接访问节点元数据，根据**表1**中定义的DICOM标签，逐一提取信息。  
4. **数据转换与对象填充**: 提取的原始字符串数据将被转换为PatientData对象所需的正确类型（如计算年龄、格式化日期），然后填充到TAVRStudySession持有的PatientData实例中。  
5. **UI更新**: PatientData对象更新后，其内容将被映射并显示在UI的患者信息表中，供用户审核。

### **3.3. 瓣膜特异性配置**

为了实现瓣膜选择的灵活性和可维护性，插件将包含一个外部的、易于编辑的配置文件（例如valve\_config.json）。该文件定义了支持的瓣膜品牌及其对应的型号列表。

JSON

{  
  "Medtronic":,  
  "Edwards Lifesciences":,  
  "Venus Medtech": \["VenusA-Valve", "VenusA-Plus"\],  
  "MicroPort": \["VitaFlow"\],  
  "Peijia Medical":  
}

插件启动时会读取此文件，用其内容填充“瓣膜品牌”下拉菜单。当用户选择一个品牌时，程序会根据该品牌查询此配置，并用相应的型号列表更新“瓣膜型号”下拉菜单。这种设计使得未来添加新瓣膜支持时，只需修改此JSON文件，而无需更改任何Python代码。

### **3.4. 事件处理与工作流逻辑**

整个数据加载验证流程可以通过一个序列图（Sequence Diagram）清晰地描述：

1. **用户**点击插件启动图标。  
2. **系统**创建并显示DataLoadingDialog。  
3. **用户**点击“加载4D DICOM序列”按钮。  
4. **DataLoadingDialog**调用slicer.DICOMBrowser。  
5. **用户**在DICOMBrowser中选择一个序列并确认。  
6. **DataLoadingDialog**接收加载完成信号，并获取加载的节点。  
7. **DataLoadingDialog**执行validate\_sequence\_node()，确认其为vtkMRMLSequenceNode。  
8. 若验证成功，**DataLoadingDialog**调用parse\_dicom\_metadata()。  
9. parse\_dicom\_metadata()从节点中读取DICOM标签，更新TAVRStudySession中的PatientData对象。  
10. **DataLoadingDialog**根据更新后的PatientData刷新UI显示。  
11. **用户**在UI中审核、编辑数据，并选择瓣膜品牌和型号。  
12. **DataLoadingDialog**的内部逻辑检测到所有必填项均已完成，激活“确认并继续”按钮。  
13. **用户**点击“确认并继续”。  
14. **DataLoadingDialog**关闭，主插件UI（包含心动周期管理面板）被激活，工作流进入下一阶段。

## **四、组件级设计：心动周期管理**

### **4.1. 用户界面规格**

在数据加载和验证成功后，插件的主UI将展示一个专用于心动周期管理的核心面板。该面板的设计旨在提供对4D数据的直观、精确的控制。

* **核心组件**:  
  1. **心动周期时间轴 (ctkSliderWidget)**: 一个水平滑块，其范围和刻度将自动映射到加载的vtkMRMLSequenceNode的帧数。滑块的每一位置都对应4D序列中的一个特定时间点。  
  2. **时相百分比显示 (QLabel)**: 位于滑块上方或下方的一个标签，实时显示当前滑块位置对应的R-R间期百分比。例如，显示为“**当前时相: 73.0%**”。这个百分比的精确显示对于满足临床研究的记录要求至关重要 1。  
  3. **关键时相标记按钮 (QPushButton)**: 两个醒目的按钮：  
     * “**标记舒张末期 (Mark End-Diastole)**”  
     * “标记收缩末期 (Mark End-Systole)”  
       这两个按钮旁边可以设置小标签，用于显示已标记时相的百分比，为用户提供即时反馈。

### **4.2. 实时视图同步**

该功能是实现交互式时相识别的基础。其技术实现依赖于Slicer的信号与槽（Signals and Slots）机制。

1. **信号连接**: 在模块初始化时，ctkSliderWidget的valueChanged(double)信号将被程序化地连接到一个自定义的槽函数（slot function），例如on\_timeline\_slider\_changed(new\_value)。  
2. **槽函数逻辑**: 该槽函数接收滑块的新值（通常是帧索引），并立即调用TAVRStudySession中持有的vtkMRMLSequenceBrowserNode实例的SetSelectedItemNumber(int)方法。  
3. **Slicer内部机制**: vtkMRMLSequenceBrowserNode在接收到SetSelectedItemNumber调用后，会自动更新其代理的vtkMRMLScalarVolumeNode，并触发Slicer的渲染管线进行重绘。  
4. **结果**: 最终效果是，当用户拖动滑块时，Slicer的所有2D切片视图（Axial, Sagittal, Coronal）和3D视图都会即时、同步地更新，流畅地展示心脏和瓣膜在整个心动周期中的动态变化过程 1。

### **4.3. 关键时相标记与持久化**

此功能的设计必须确保临床分析的**可重复性**和**可追溯性**。临床研究和标准报告 1 不仅要求在特定时相（如舒张期）进行评估，还要求精确记录该时相在心动周期中的具体位置（百分比）。因此，一个简单的“已标记”状态是远远不够的。

这一要求直接驱动了TAVRStudySession中关于时相存储的数据结构设计。它不能是一个简单的布尔值，而必须是一个能够存储精确时相信息的结构，例如一个字典：

Python

self.marked\_phases \= {  
    'end\_diastole': {'frame\_index': None, 'phase\_percent': None},  
    'end\_systole':  {'frame\_index': None, 'phase\_percent': None}  
}

事件处理逻辑:  
当用户点击“标记舒张末期”按钮时，其事件处理器将执行以下步骤：

1. **获取当前帧索引**: 调用sequence\_browser\_node.GetSelectedItemNumber()获取当前滑块所在的帧索引。  
2. **获取当前时相百分比**: vtkMRMLSequenceNode为其序列中的每一帧都存储了元数据，包括索引值（IndexValue）和单位（IndexUnit），通常单位是“%”。脚本将查询当前帧的IndexValue，得到精确的R-R间期百分比（例如，73.0）。  
3. **持久化存储**: 将获取到的帧索引和时相百分比，存储到TAVRStudySession的self.marked\_phases字典中，键为'end\_diastole'。  
4. **UI反馈**: 更新UI，在“标记舒张末期”按钮旁边的标签中显示“已标记 @ 73.0%”。

“**标记收缩末期**”按钮的逻辑完全相同，只是数据存储在'end\_systole'键下。

这种设计确保了后续模块（如模块三）在需要进行HALT（低密度瓣叶增厚）分析时，可以向TAVRStudySession查询：get\_marked\_phase('end\_diastole')。它将收到包含frame\_index和phase\_percent的完整信息。模块三随即可以：

* 使用frame\_index来设置vtkMRMLSequenceBrowserNode，确保视图跳转到正确的图像帧。  
* 使用phase\_percent来自动填充最终报告中“HALT评估”部分的“测量期相 %”字段。

这样就构建了一个从时相标记到最终报告生成的、完全自动化的、数据可追溯的闭环工作流，满足了高质量临床研究的严格要求。

| 错误情况 (Error Condition) | 触发条件 (Trigger) | 系统响应 (System Response) | 用户消息 (User Message) |
| :---- | :---- | :---- | :---- |
| **加载了无效数据类型** | 用户在DICOM浏览器中选择了一个非序列数据（如单个3D容积、2D图像）。 | 拒绝数据加载。保持数据加载对话框打开，等待用户重新选择。 | **错误对话框**: "加载失败：请选择一个4D心脏CT DICOM序列。您所选的数据并非有效的时间序列。" |
| **缺少关键DICOM标签** | 加载的DICOM序列缺少如PatientID (0010,0020)等关键元数据。 | 正常加载图像数据，但在UI的患者信息表中，将对应的字段留空并以红色边框或背景高亮显示，以提示用户手动输入。 | **状态栏/日志警告**: "警告：无法在DICOM数据中找到'受试者编号'。请务必手动输入。" |
| **强制性输入不完整** | 用户在未选择瓣膜品牌和型号的情况下，试图点击“确认并继续”按钮。 | “确认并继续”按钮保持禁用状态。当鼠标悬停在按钮上时，显示一个工具提示（Tooltip）。 | **工具提示**: "请选择瓣膜品牌和型号以继续分析。" |
| **时间序列不一致** | 加载的4D序列具有不规则的时间步长或缺失的帧。 | 允许加载数据，但在加载完成后显示一个非阻塞性的警告对话框或在状态栏显示持久性警告。 | **警告对话框**: "警告：此序列的心动周期时相信息不规则，可能导致时相百分比计算不准确。请谨慎操作。" |
| **用户中途取消** | 用户在数据加载对话框中点击“取消”按钮。 | 立即关闭对话框，并重置TAVRStudySession的状态，清除任何已加载的临时数据，返回到插件的初始界面。 | 无需消息。 |
| **重复加载数据** | 用户在已加载一个序列的情况下，再次点击“加载4D DICOM序列”按钮。 | 弹出一个确认对话框，询问用户是否要放弃当前会话并加载新数据。 | **确认对话框**: "您确定要放弃当前分析会话并加载新的数据吗？所有未保存的进度将会丢失。" |

## **六、模块API与接口**

本部分正式定义了模块一将向插件中的其他模块暴露的公共应用程序接口（API）。这个接口是模块间通信的“契约”，对于实现模块化开发、并行工作以及长期的代码可维护性至关重要。所有API都将作为TAVRStudySession类（或其管理者）的公共方法提供。

### **TAVRStudySession 公共API定义**

* get\_patient\_data() \-\> PatientData:  
  * **描述**: 返回包含所有患者及研究信息的PatientData对象实例。  
  * **返回值**: 一个PatientData类的实例。  
  * **调用场景**: 最终报告生成模块（模块四）调用此方法以获取所有需要填入报告表单的文本信息。  
* get\_volume\_sequence\_node() \-\> vtkMRMLSequenceNode:  
  * **描述**: 返回代表已加载的4D CT数据的主vtkMRMLSequenceNode。  
  * **返回值**: 一个vtkMRMLSequenceNode对象。  
  * **调用场景**: 模块二（分割）和模块三（测量）需要访问原始的四维图像数据时调用此方法。  
* get\_sequence\_browser\_node() \-\> vtkMRMLSequenceBrowserNode:  
  * **描述**: 返回用于控制序列播放和帧选择的vtkMRMLSequenceBrowserNode。  
  * **返回值**: 一个vtkMRMLSequenceBrowserNode对象。  
  * **调用场景**: 任何需要程序化地跳转到特定时间点的模块（如模块三在进行HALT分析时跳转到舒张末期）都会调用此方法。  
* get\_selected\_valve() \-\> dict:  
  * **描述**: 返回一个包含用户所选瓣膜品牌和型号的字典。  
  * **返回值**: 一个字典，格式为 {'brand': str, 'model': str}。  
  * **调用场景**: 模块二和模块三在执行任何与瓣膜相关的几何分析前，必须调用此方法以获取配置，并据此选择正确的分割或测量算法。  
* get\_marked\_phase(phase\_name: str) \-\> dict:  
  * **描述**: 根据指定的时相名称（'end\_diastole' 或 'end\_systole'），返回该时相的详细信息。  
  * **参数**: phase\_name \- 字符串，必须是 'end\_diastole' 或 'end\_systole'。  
  * **返回值**: 一个字典，格式为 {'frame\_index': int, 'phase\_percent': float}。如果该时相未被标记，则返回None或字典中的值为None。  
  * **调用场景**: 模块三在进行特定时相的测量时（如HALT, RELM），调用此方法来获取正确的帧索引和用于报告的百分比值。  
* is\_ready() \-\> bool:  
  * **描述**: 检查模块一的工作流程是否已全部完成，即是否可以安全地进入下一分析阶段。  
  * **返回值**: 布尔值。当且仅当一个有效的4D序列已加载，并且所有强制性输入（如瓣膜型号）都已完成后，返回True。  
  * **调用场景**: 插件的主工作流控制器在激活后续模块的UI或功能前，会调用此方法进行前置条件检查。

通过这套清晰、明确的API，模块一有效地将其内部复杂的实现细节（如DICOM解析、UI事件处理）封装起来，仅对外提供一个简洁、稳定的接口。这种设计大大降低了模块间的耦合度，使得整个插件系统更加健壮、易于测试和扩展。
